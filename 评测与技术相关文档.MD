## 一、评测集来源（按任务类型分类）

### 1.1 简历解析评测集

**ResumeBench（首推）**

- **来源**：EMNLP 2025 会议论文《Beyond Human Labels: A Multi-Linguistic Auto-Generated Benchmark for Evaluating Large Language Models on Resume Parsing》
- **规模**：2,500 份合成简历，覆盖 50 种模板、30 个职业领域、5 种语言（含中文）
- **生成方式**：人机协同（Human-in-the-loop）管道生成，兼顾真实性、多样性与隐私合规
- **评测覆盖**：已评测 24 个 SOTA LLM（含 GPT-4o），发现模型在跨语言结构对齐和语义歧义处理上存在显著挑战
- **获取地址**：ACL Anthology 页面
- **OfferFit 适用场景**：L1 自动化评测中的“字段抽取 F1”和“文件格式成功率”测试

**Layout-Aware Resume Parsing Benchmark**

- **来源**：arXiv 2025《Layout-Aware Parsing Meets Efficient LLMs: A Unified, Scalable Framework for Resume Information Extraction and Evaluation》
- **特点**：布局感知框架，结合微调布局解析器 + 并行提示 LLM + 自动化评测框架
- **适用场景**：测试 PDF/Word 多格式解析能力，尤其是复杂排版简历

**Kaggle Resumes for Parsing Benchmarking**

- **来源**：Kaggle 公开数据集（2026）
- **特点**：基于 LLM + 模板混合生成的合成简历，附带可控的 Ground Truth 标注
- **适用场景**：快速构建最小评测集的补充数据源

**Libelle Golden Set**

- **来源**：GitHub 开源项目
- **特点**：10 份合成简历 + 对应的“Golden JSON”标注文件，强调清晰性和可调试性
- **适用场景**：Parser 的冒烟测试和快速回归

### 1.2 简历-JD 匹配评测集

**RJDB（Resume-Job Description Benchmark）——核心推荐**

- **来源**：Megagon Labs × Indeed 联合发布，论文《Distilling Large Language Models using Skill-Occupation Graph Context for HR-Related Tasks》
- **规模**：超过 50,000 组三元组（JD + 匹配简历 + 不匹配简历）
- **构建方法**：基于技能-职业图谱（Skill-Occupation Graph）蒸馏 GPT-4 的领域知识生成
- **实验结果**：学生模型在 RJDB 上训练后达到接近/超越教师模型（GPT-4）的性能
- **代码仓库**：https://github.com/megagonlabs/rjdb
- **⚠️ 重要说明**：数据集部分内容由 GPT-4 生成，遵循 OpenAI 使用政策
- **OfferFit 适用场景**：L1“技能覆盖 F1”、L2“匹配推理能力”的核心测试集

**PJB（Person-Job Benchmark）**

- **来源**：CareerInternational 研究团队，arXiv 2026
- **定位**：推理感知（Reasoning-Aware）的检索评测数据集
- **规模**：覆盖 6 个行业领域，近 200,000 份简历
- **核心创新**：不追求“谁分数更高”，而是回答“系统在哪里失败、为什么失败”
- **诊断维度**：显性约束验证（平行推理）+ 技能迁移推断（串行推理）
- **适用场景**：测试模型在复杂推理场景下的表现，作为 Edge Set 的补充

**JobResQA**

- **来源**：Avature × UPC 联合发布，arXiv 2026
- **规模**：581 个 QA 对，覆盖 105 组合成简历-JD 对
- **语言**：英语、西班牙语、意大利语、德语、中文（5 种语言）
- **问题层级**：3 级复杂度——从基础事实提取到跨文档复杂推理
- **偏见研究**：通过占位符控制人口统计学和职业属性，支持系统性偏见研究
- **代码仓库**：https://github.com/Avature/jobresqa-benchmark
- **适用场景**：多语言能力测试 + 公平性研究

**Kaggle Resume-JD Matching Dataset**

- **来源**：Kaggle（2026）
- **规模**：500 组合成简历-JD 对，覆盖 10 个领域
- **领域覆盖**：软件工程、数据科学、产品管理、市场营销、金融、HR、UI/UX 设计、网络安全、DevOps、销售
- **适用场景**：快速构建多领域覆盖的最小评测集

**TalentCLEF 2026**

- **来源**：CLEF 2026 评测任务
- **任务 A**：上下文化的人岗匹配（Contextualized Job-Person Matching）——对给定 JD 排序最合适的候选人简历
- **语言**：英语和西班牙语双语
- **标注方式**：人类专家标注简历与 JD 的匹配度
- **适用场景**：作为外部基准，验证 OfferFit 与 SOTA 的差距

### 1.3 技能抽取评测集

**Skill Extraction Benchmark**

- **来源**：Zenodo / Idiap 研究所
- **内容**：从简历语料中抽取的硬技能实体集合
- **特点**：多人标注同一份简历以建立可靠的 Ground Truth 共识
- **适用场景**：测试技能抽取的 Precision/Recall

**6 个统一化技能抽取数据集**

- **来源**：《Rethinking Skill Extraction in the Job Market Domain using LLMs》
- **覆盖**：4 种语言（英语、法语、德语、丹麦语），多个领域和技能类别
- **适用场景**：跨语言技能抽取能力测试

### 1.4 公平性与偏见评测集

**FAIRE（Fairness Assessment In Resume Evaluation）——核心推荐**

- **来源**：arXiv 2025
- **方法**：直接评分（Direct Scoring）+ 排序（Ranking）两种方式
- **核心机制**：微调简历中的种族或性别标识，测量模型评分变化
- **发现**：所有模型均存在不同程度的偏见，但幅度和方向差异显著
- **代码仓库**：https://github.com/athenawen/FAIRE-Fairness-Assessment-In-Resume-Evaluation
- **OfferFit 适用场景**：Tier 1 门禁中的“偏见歧视”检测（G-03）

**FairCVdb**

- **来源**：arXiv 论文
- **规模**：24,000 份合成简历档案
- **包含信息**：性别 + 种族（2 个人口统计学属性）、职业、面部图像、姓名、7 个简历属性、短传记
- **特点**：多模态（文本 + 图像 + 结构化数据）公平性研究
- **适用场景**：多模态公平性测试的补充数据源

**FairCV（中文版）**

- **来源**：GitHub 开源项目
- **定位**：中文互联网求职简历数据集生成脚本——偏见检测与 AI 筛选研究
- **覆盖维度**：性别、年龄、地域等多维度偏见
- **⚠️ 特别重要**：这是**唯一专门针对中文语境**的简历偏见检测数据集
- **OfferFit 适用场景**：中文场景下的公平性门禁测试

**PopResume**

- **来源**：arXiv
- **定位**：人口统计学代表性的简历数据集，用于 LLM/VLM 简历筛选器的因果公平性审计
- **创新点**：基于人口统计数据而非人工注入的人口统计学信息
- **适用场景**：高级公平性研究

### 1.5 职业生涯轨迹数据集

**JobHop**

- **来源**：arXiv，基于比利时 VDAB 公共就业服务匿名化简历
- **特点**：大规模公开的职业轨迹数据集
- **适用场景**：职业规划与路径预测能力的测试

**Career Path Prediction Dataset**

- **来源**：arXiv 2023
- **规模**：2,164 份匿名化职业历史，带 ESCO 职业标签标注
- **适用场景**：长期职业规划建议的测试

## 二、理论体系

### 2.1 评测哲学基础

**从“谁更强”到“哪里失败”的范式转移**
PJB 论文提出核心观点：检索模型趋同后，关键问题不再是“谁分数更高”，而是“系统在哪里失败、为什么失败”。这一思想直接支撑了 OfferFit V2.0/V3.0 从“总分导向”转向“短板导向”的设计。

**木桶原理（短板理论）的数学化**
在复合型任务中，系统质量由其最弱环节决定。V2.0 采用 `min(dimensions)` 作为聚合分数的数学依据，避免维度间的补偿谬误。

**悲观主义聚合（Pessimistic Aggregation）**
在高风险决策场景（招聘筛选）中，应采用保守估计而非乐观平均。这与 PJB 强调的“诊断性评测”一脉相承。

### 2.2 匹配推理理论

**平行推理（Parallel Reasoning） vs 串行推理（Serial Reasoning）**
PJB 框架将人岗匹配分解为：

- **平行推理**：独立验证显性约束（地点、学历、年限、薪资、关键词）并综合结果
- **串行推理**：围绕岗位胜任力进行多跳语义抽象——将职责映射到隐性技能要求、将跨行业经验解读为可迁移能力

**技能-职业图谱（Skill-Occupation Graph）**
RJDB 的构建依赖技能-职业图谱来确保数据多样性和生成上下文。图谱提供了技能与职业之间的结构化关系，是 JD 拆解和技能匹配的理论基础。

### 2.3 公平性理论

**统计性歧视理论**
FairCV 项目将统计性歧视理论引入 LLM 偏见研究——模型在训练数据中学习到的群体统计差异可能转化为系统性歧视。

**因果公平性（Causal Fairness）**
PopResume 采用路径特定效应（Path-Specific Effect, PSE）进行公平性评估，超越了简单的“结果差异”统计，关注歧视的因果机制。

**EU AI Act 合规框架**
JobResQA 明确指出，HR 场景的 AI 应用需遵循 EU AI Act 对公平性、可解释性和负责任使用的要求。

### 2.4 LLM-as-Judge 理论

**裁判可靠性三要素**
综合综述论文指出，可靠的 LLM-as-Judge 需关注：

- **鲁棒性（Robustness）**：对输入扰动不敏感
- **敏感性（Sensitivity）**：能区分质量差异
- **自偏好（Self-preference）**：避免给同类模型打高分

**多模型盲审制**
Agent-as-a-Judge 综述指出，从单模型裁判演进到多智能体辩论框架是提升裁判可靠性的关键路径。

## 三、技术实现

### 3.1 技术选型建议


| 组件               | 推荐方案                                   | 依据                                               |
| :----------------- | :----------------------------------------- | :------------------------------------------------- |
| **多智能体框架**   | LangGraph                                  | 支持有向循环图，适合多角色协作【CLAUDE.md 指定】   |
| **LLM 网关**       | LiteLLM                                    | 统一接入 OpenAI/Anthropic/DeepSeek，便于多模型盲审 |
| **事实一致性检查** | GPT-4o-mini + NLI 模型                     | 轻量 + 高准确率                                    |
| **裁判模型**       | DeepSeek-V3（开源）+ Claude/Gemini（闭源） | 避免自偏好【V3.0 设计】                            |
| **解析引擎**       | 微调 Layout Parser + 并行提示 LLM          | 处理复杂排版                                       |
| **评测框架**       | 自建 Python 脚本                           | 灵活性 + 可控性                                    |

### 3.2 关键算法

**ELO 排名算法**
用于盲测竞技场模式，通过两两比较更新模型积分，消除绝对分数的模型偏见。

**对比学习（Contrastive Learning）**
ConFit 方法将简历-JD 数据集建模为稀疏二分图，通过数据增强和对比学习提升匹配效果。

**NLI 蕴涵检查**
用自然语言推理模型判断“简历原文句子”是否蕴涵“改写后句子”，替代脆弱的正则表达式。

## 四、OfferFit 评测体系落地指南

### 4.1 数据集构建优先级（Phase 1）

```
Week 1: Smoke Set（10份）
  └── Libelle Golden Set（开源） + 2份极简自建样本

Week 2-3: Core Set（100份） 
  └── 50份脱敏真实简历（内部收集）+ 50份对应JD
  └── 覆盖：产品、运营、技术、设计、市场

Week 4: Edge Set（50份）
  └── 构造高难度样本：乱码PDF、时间空档、转行、职级矛盾
  └── 补充：FAIRE 公平性样本 + FairCV 中文样本
```

### 4.2 评测流水线架构

```
输入（简历 + JD）
    ↓
Tier 1: 安全与事实门禁
    ├── G-01 虚构检查（NLI / GPT-4o-mini）
    ├── G-02 敏感信息泄露
    ├── G-03 偏见检测（FAIRE / FairCV 样本）
    ├── G-04 硬性条件欺诈
    └── G-05 严重问题漏报
    ↓（全部 PASS）
Tier 2: 能力剖面图（五维平权）
    ├── P1 输入保真度（ResumeBench / 自建标注）
    ├── P2 语义洞察力（RJDB 子集）
    ├── P3 逻辑推理力（PJB 诊断标签）
    ├── P4 生成执行力（自建改写样本）
    └── P5 交互稳健性（Edge Set）
    ↓
输出：木桶分 + 波动指数 + ELO排名
```

### 4.3 关键指标阈值参考


| 指标             | 阈值   | 数据来源依据     |
| :--------------- | :----- | :--------------- |
| 字段抽取 F1      | > 0.85 | ResumeBench 基准 |
| 技能覆盖 F1      | > 0.80 | RJDB 实验        |
